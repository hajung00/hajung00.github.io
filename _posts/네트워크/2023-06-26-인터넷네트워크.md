## 1. 인터넷 통신(클라이언트와 서버)

클라이언트와 서버는 인터넷망을 통해 통신한다.
인터넷망의 수많은 노드를 거쳐 클라이언트에서 서버로 이동한다.

?? 어떤 규칙으로 어떻게 넘어가는가??

## 2. IP(인터넷 프로토콜)

IP주소를 통해 가능함.

클라이언트 IP주소 ----------> 서버 IP주소

-역할

1. 지정한 IP주소(IP address)에 데이터 전달
2. 패킷(Packet)이라는 통신 단위로 데이터 전달

- 패킷(Packet) 정보
  메시지만 보내는 것이 아니라 패킷 규칙에 맞게 작성해야함

- 출발지IP, 목적지IP, 전송데이터(메시지)

상황 1) 클라이언트 패킷 전달 -그림 첨부

상황 2) 서버 패킷 전달 -그림 첨부

- IP 프로토콜의 한계

1. 비연결성
   - 패킷을 받을 대상이 없거나 서비스 불능 상태여도 패킷 전송
2. 비신뢰성
   - 중간에 패킷이 사라지면?
   - 패킷이 순서대로 안오면?
3. 프로그램 구분
   - 같은 IP를 사용하는 서버에서 통신하는 애플리케이션이 둘 이상이면?

!! TCP로 해결

3. TCP UDP

- IP의 문제점을 해결해주는 것이 TCP!

!TCP 알아보기전 복습

- 인터넷 프로토콜 스택의 4계층
  애플리케이션 계층 - HTTP,FTP
  전송 계층 - TCP, UDP
  인터넷 계층 - IP
  네트워크 인터페이스 계층

- 프로토콜 계층

* 그림 첨부

* TCP 특징

1. 연결지향 - TCP 3way handshacke(가상 연결)

- 그림 첨부

1-1. 클라이언트가 서버에게 접속 요청(SYN)
1-2. 서버에서 요청 수락(ACK) 후 접속 요청(SYN)
1-3. 클라이언트가 서버의 요청 수락(ACK)
1-4. 데이터 전송

- 따라서 서로를 믿을 수 있음!

2. 데이터 전달 보증: 데이터의 전달 여부를 알 수 있음.

- 그림 첨부
  서버에서 데이터를 잘 전달 받았으면 클라이언트에게 응답 보내줌.

3. 순서 보장

- 그림 첨부

* 이 모든것은 TCP/IP패킷 정보에 전송 제어, 순서, 검증 정보가 담겨 있기에 가능한 것이다!

4. UDP

- IP와 거의 같다. +PORT, +체크섬 정도만 추가
- 한 컴퓨터에서 여러 애플리케이션을 사용하는 경우(게임, 음악 플레이어) 한 IP로 여러 패킷이 옴.
  구분하는 방법 = PORT 사용
  ex) 4000port => 게임 패킷, 5000port => 음악 플레이어 패킷
