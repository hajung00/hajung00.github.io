## 1. 인터넷 통신(클라이언트와 서버)

클라이언트와 서버는 인터넷망을 통해 통신한다.
인터넷망의 수많은 노드를 거쳐 클라이언트에서 서버로 이동한다.

?? 어떤 규칙으로 어떻게 넘어가는가??

## 2. IP(인터넷 프로토콜)

IP주소를 통해 가능함.

클라이언트 IP주소 ----------> 서버 IP주소

-역할

1. 지정한 IP주소(IP address)에 데이터 전달
2. 패킷(Packet)이라는 통신 단위로 데이터 전달

- 패킷(Packet) 정보
  메시지만 보내는 것이 아니라 패킷 규칙에 맞게 작성해야함

- 출발지IP, 목적지IP, 전송데이터(메시지)

상황 1) 클라이언트 패킷 전달 -그림 첨부

상황 2) 서버 패킷 전달 -그림 첨부

- IP 프로토콜의 한계

1. 비연결성
   - 패킷을 받을 대상이 없거나 서비스 불능 상태여도 패킷 전송
2. 비신뢰성
   - 중간에 패킷이 사라지면?
   - 패킷이 순서대로 안오면?
3. 프로그램 구분
   - 같은 IP를 사용하는 서버에서 통신하는 애플리케이션이 둘 이상이면?

!! TCP로 해결

## 3. TCP UDP

#### 3-1. TCP

- IP의 문제점을 해결해주는 것이 TCP!

!TCP 알아보기전 복습

- 인터넷 프로토콜 스택의 4계층
  애플리케이션 계층 - HTTP,FTP
  전송 계층 - TCP, UDP
  인터넷 계층 - IP
  네트워크 인터페이스 계층

- 프로토콜 계층

* 그림 첨부

* TCP 특징

1. 연결지향 - TCP 3way handshacke(가상 연결)

- 그림 첨부

1-1. 클라이언트가 서버에게 접속 요청(SYN)
1-2. 서버에서 요청 수락(ACK) 후 접속 요청(SYN)
1-3. 클라이언트가 서버의 요청 수락(ACK)
1-4. 데이터 전송

- 따라서 서로를 믿을 수 있음!

2. 데이터 전달 보증: 데이터의 전달 여부를 알 수 있음.

- 그림 첨부
  서버에서 데이터를 잘 전달 받았으면 클라이언트에게 응답 보내줌.

3. 순서 보장

- 그림 첨부

* 이 모든것은 TCP/IP패킷 정보에 전송 제어, 순서, 검증 정보가 담겨 있기에 가능한 것이다!

#### 3-2.UDP

- IP와 거의 같다. +PORT, +체크섬 정도만 추가

## 4. PORT

- 그림 첨부
- 한 PC가 게임, 화상 통화, 웹브라우저 요청 등 여러 서버와 통신 해야함.
- 서버에서 PC로 패킷을 전송하는데 어느 서버에서 온 패킷인지 알 수 없음.

! PORT를 사용하여 구분 !

- 그림 첨부
  => PORT는 같은 IP내에서 프로세스를 구분할 때 사용!
  ex) 게임 서버에 연결하려면 IP: 200.200.200.2의 11220번 PORT에 연결 할 것을 패킷에 정보를 넣어 전송

HTTP => PORT 80 , HTTPS => PORT 443

## 5. DNS(Domain Name System)

DNS란?
도메인 명을 IP 주소로 변환해 주는 시스템

- IP의 복잡성
- 그림 첨부

- IP변경 가능성
- 그림 첨부

!! DNS로 이를 해결 !!

-그림 첨부

- 도메인 구매 후 사용 가능
- DNS 서버가 도메인 명에 맞는 IP주소를 응답해준다.
- IP보다 도메인이 기억하기 더 쉬움.
- IP변경 시, DNS서버에 IP변경하면 변경 가능성 문제를 해결 할 수 있음.
