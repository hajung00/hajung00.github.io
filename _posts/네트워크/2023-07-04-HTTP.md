## 1. HTTP

- 특징

1. 클라이언트 서버 구조

-그림 첨부

- request response 구조
- 클라이언트는 서버에 요청을 보내고, 응답을 대기
- 서버가 요청에 대한 결과를 만들어서 응답

## 2. 무상태 프로토콜(스테이스리스)

- stateful, stateless 차이

* stateful(상태 유지)
  서버가 클라이언트의 이전상태(문맥)를 보존 => 항상 같은 서버가 유지되어야함.
* 그림 첨부
  상황 a) 서버가 중간에 문제가 생김
* 그림 첨부
* 장애 발생

* stateless(무상태)
  서버가 클라이언트의 이전상태(문맥)를 보존하지 않음
  (클라이언트에서 그때 그때 필요한 데이터를 다 담아서 보내줌)
* 그림 첨부
  상황 a) 서버가 중간에 문제가 생김
* 그림 첨부
* 다른 서버로 교체하여 장애 생기지 않음.

* 차이점
  서버가 중간에 변경되었을 때, stateful에서 장애 발생 stateless는 장애 없음.

- stateless의 장단점
  장점: 서버 확장성 높음(클라이언트 요청이 증가해도 서버를 대거 투입 가능, 무한한 서버 증설 가능)
  단점: 클라이언트가 추가 데이터 전송

- stateless 실무 한계

* 모든 것을 무상태로 설계 할 수 있는 경우도 있고 없는 경우도 있다.
* 무상태 (ex) 로그인이 필요 없는 단순한 서비스 소개 화면)
* 상태 유지(ex) 로그인)
* 로그인한 사용자의 경우 로그인 했다는 상태를 서버에 유지
* 일반적으로 브라우저 쿠키와 서버 세션등을 사용해서 상태 유지
* 상태 유지는 최소한만 사용

## 3. 비연결성

- 연결을 유지하는 모델

* 그림 첨부
* 클라이언트가 요청을 하지 않는 상태에도 서버와의 연결을 유지

- 연결을 유지하지 않는 모델

* 그림 첨부
* 서버의 자원을 요청과 응답 주고받을 때만 사용할 수 있음.
* 최소한의 자원 사용
* HTTP의 기본이 비연결성

* 단점
* TCP/IP 연결을 새로 맺어야 함 -3 way handshake 시간 추가
* 웹 브라우저로 사이트를 요청하면 필요한 문서(HTML, CSS, JS, 이미지)등 수 많은 자원이 함께 다운로드
* 지속 연결로 문제 해결

- 지속 연결

* 그림 첨부

## 4. HTTP 메시지

- HTTP 메시지 구조
- 그림 첨부
- 시작 라인 = 요청 라인, 상태 라인
- 헤더:field-name ":" OWS field-value OWS (OWS: 띄어쓰기 허용)

* 용도( HTTP 전송에 필요한 모든 부가 정보: 메시지 바디 내용, 크기, 압축, 인증, 브라우저 정보, 캐시 등.,.)

- 바디: 실제 전송할 데이터(byte로 표현할 수 있는 모든 데이터 전송 가능)

- HTTP 요청 메시지
- 그림 첨부
- 시작 라인(요청 라인): HTTP 메서드(get, put...), 요청 대상(path), HTTP 버전

- HTTP 응답 메시지
- 그림 첨부
- 시작 라인(상태 라인): HTTP 버전, HTTP 상태코드(요청 성공, 실패 여부), 이유 문구(사람이 이해할 수 있는 짧은 상태 코드 설명 글)
